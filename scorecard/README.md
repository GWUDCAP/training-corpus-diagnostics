# Public Scorecard

The scorecard is a public registry of corpus positions under the fixed Training Corpus Diagnostics metric suite. Entries are diagnostic positions, not quality rankings and not claims about downstream model behavior.

Submissions are PR-gated. Contributors generate a full artifact bundle locally, host that bundle externally, and commit only a small scorecard entry to this repository. Maintainers and reviewing agents can then validate schema, hashes, provenance, and interpretation language without bloating the repo.

## Status Tiers

- `submitted`: PR opened, not validated.
- `artifact_validated`: schema, artifact hash, settings, and provenance fields pass automated or maintainer checks.
- `agent_reviewed`: a maintainer or reviewing agent has checked provenance and interpretation language.
- `reproduced`: a maintainer or trusted runner reran the public metric suite and reproduced the submitted summaries within release tolerance.

The registry lives under `scorecard/registry/`:

- `index.json`: public entry index.
- `entries/<label>.json`: small per-corpus scorecard entry files.

Full chunks, embeddings, provenance sidecars, and matrix outputs should live in an externally hosted artifact bundle referenced by URL and SHA-256 digest.

## Contributor Workflow

1. Prepare a source config by copying `configs/example_local_sources.json` or writing an equivalent Hugging Face source recipe.
2. Build chunks with `scripts/build_corpus_artifacts.py build-chunks`.
3. Build embeddings with `scripts/build_corpus_artifacts.py build-embeddings`.
4. Run the matrix with `scripts/run_corpus_matrix.py`.
5. Package a submission with `scripts/package_scorecard_submission.py`.
6. Host the generated zip somewhere reviewers can access and compute its SHA-256 digest.
7. Open a PR adding `scorecard/registry/entries/<label>.json` and updating `scorecard/registry/index.json`.

Canonical example:

```bash
python scripts/build_corpus_artifacts.py build-chunks \
  --sources my_sources.json \
  --out-root my_scorecard_run

python scripts/build_corpus_artifacts.py build-embeddings \
  --manifest my_scorecard_run/corpus_manifest.generated.json \
  --out-root my_scorecard_run \
  --model-name Alibaba-NLP/gte-base-en-v1.5

python scripts/run_corpus_matrix.py \
  --manifest my_scorecard_run/corpus_manifest.generated.json \
  --out-dir my_scorecard_run/matrix

python scripts/package_scorecard_submission.py \
  --manifest my_scorecard_run/corpus_manifest.generated.json \
  --matrix-dir my_scorecard_run/matrix \
  --labels my_new_corpus \
  --source-config my_sources.json \
  --artifact-url https://example.org/my_new_corpus_scorecard_submission.zip \
  --artifact-sha256 <sha256-of-zip> \
  --out-dir my_new_corpus_scorecard_submission \
  --zip
```

Copy `my_new_corpus_scorecard_submission/scorecard_entry.json` to `scorecard/registry/entries/my_new_corpus.json`, update `scorecard/registry/index.json`, then run:

```bash
python scripts/validate_scorecard_submission.py --registry scorecard/registry
```

## Inclusion Criteria

Public entries must include:

- metric suite version `lens_effects.release.v0.1.0`;
- 2048 chunks, 768 maximum chunk characters, and 192 retained minimum characters for canonical entries;
- GTE embeddings from `Alibaba-NLP/gte-base-en-v1.5` with dimension 768;
- source, split, field, source type, extraction mode, and license/provenance notes where available;
- artifact URL and lowercase SHA-256 digest for the full submission bundle;
- standalone summaries and pair summaries generated by the public scripts;
- diagnostic-position language only.

Unknown or newly introduced corpora should use `role: "candidate"` and `stage_role: "unlabeled"` unless their stage role is externally established. Candidate pairs are positioned relative to the retained panel without changing the retained broad/targeted headline summaries.

## Agent Review Checklist

Reviewing agents should answer:

1. Does the entry pass `scripts/validate_scorecard_submission.py`?
2. Does the artifact URL resolve, and does the artifact SHA-256 match?
3. Are source, extraction mode, chunk budget, embedding model, and provenance disclosed?
4. Are chunks and embeddings row-aligned in the artifact bundle?
5. Is the entry interpretable as a corpus diagnostic rather than a quality ranking?
6. Should the PR be merged at `artifact_validated`, promoted to `agent_reviewed`, or held for reproduction?
